# 设计评审 (Adversarial Critique)

## Inconsistencies

- MOOD-VARIANT NAMING SPLIT BRAIN. The copy system buckets mood into 5 bands (极好/好/平/低/极低) but the art pipeline's mood files are a 5-name set (idle|happy|eating|sad|sleeping) and the soft-fail spec uses STATE flags (SICK/SULKING/HIDING/LONELY/饿/脏/困). There is no defined mapping from mood_band -> sprite mood file, nor from state_flag -> sprite. The /api/action response promises an `animation` string ('eat_happy'|'refuse_pout'|'sulk'|'hide'|'reunion') but the art matrix only authors idle+happy+sad+sleeping (eating deferred); 'refuse_pout', 'sulk', 'hide', 'reunion' animations have NO corresponding authored sprite or derived variant. The animation-cue vocabulary and the sprite-file vocabulary were designed independently and never reconciled.
- STAT-SET MISMATCH between tech schema and core systems. Core spec §0 lists `feed_count_today` resetting at user-local 04:00; the schema has `feed_count_day` + `snack_count_day` + `pet_count_day` resetting via `daily_reset_on` DATE — but DATE has no time-of-day, so the '04:00 reset' rule from the core spec cannot be expressed by the schema column as designed. The 04:00 boundary is silently dropped to a plain local-date rollover (midnight).
- STAGE CAP CONTRADICTION. Growth table sets egg caps at 80 and start satiety at 70/energy 80/cleanliness 80/health 90 — but health starts at 90 which EXCEEDS the egg cap of 80. recompute() clamps to cap_for_stage, so a freshly hatched egg's health is instantly clamped 90->80 on the very first read. Same for energy/cleanliness at 80 (==cap, borderline) but health 90>80 is a definite contradiction between the start values and the egg cap.
- QUIZ ANCHOR vs BESTIARY axisProfile DRIFT. The quiz anchors encode echo_fox as express -5 (理性), but the personality model lists echo_fox as 理性 HIGH which is consistent; however puff_seal is axisProfile '安稳 MID' (curio ~0) yet bestiary text says it likes 滑行/顶球 (active) and dislikes 太激烈吵闹的活动 — and its anchor curio=0 places it as the centroid magnet (see scopeRisks). clay_golem anchor express=-3 but archetype profile says '撒娇 LOW-MID (理性偏憨)'; stone_egg express=-6 vs clay -3 are only 3 apart on the axis the design claims is 'most differentiating' — they will frequently collide and resolve by tiebreak, undercutting the 'fated' feel for the 安稳/理性 cluster.
- SLEEP WINDOW DEFINITION IS CIRCULAR/UNDERSPECIFIED. Core §2 says a pet is 'asleep' only when user explicitly 哄睡s OR during [23:00,07:00) IF last interaction was a 哄睡 that night. But energy regen (+9/h while asleep) and the natural 'it naps itself at night' recovery in §4 (困 state) contradict each other: §4 says 'or it naps itself at night' implying passive regen, but §2 gates asleep on an explicit 哄睡. An engineer cannot tell whether energy passively regenerates overnight without a 哄睡 action.
- DECAY MODIFIER STACKING UNDEFINED. §1 lists FOUR multiplicative decay modifiers (bond comfort, sleep, sickness, stage) but never states they all multiply together, nor the order, nor whether bond-comfort applies while asleep. A sick + asleep + high-bond baby has 4 multipliers on satiety; the combined effective rate is undefined and could even invert intent (e.g. sleep ×0.5 × sick ×1.4 × stage ×1.15 × bond ×0.7).
- REVEAL 'QUOTE-BACK' DEPENDS ON DATA THE QUIZ DOESN'T GUARANTEE. §4.3 mirror tables are keyed to specific answers (Q1-A, Q3-A, Q5-A for mochi). But a user matched to mochi_pudding need not have picked ANY of those answers — they could have reached mochi via a totally different answer vector. The 'pick the highest-amplitude scored option and look it up in this archetype's mirror table' will MISS for most users (the mirror table only has ~3 entries per archetype vs 24 possible scored options), falling back to a generic line — gutting the 'it's literally me' jolt that the whole acquisition fantasy rests on.
- TECH ARCH INVENTS STATUS-CHECK APIs THAT DON'T EXIST. The cloudpet.ts freshness check uses `withDb('cloudpet', async (q)=>...)` and a `Check` type with `{ok, warn, detail}`. The actual status repo uses `CheckFn` returning a single `{id,group,name,status:'ok'|'warn'|'fail',detail}` object and `query(db, sql)` — not `withDb`, not a `Check[]` return, no boolean `ok`/`warn` fields. cron.ts uses `{id,name,schedule,warnAfterMs,failAfterMs}` not `{group,job,label,maxAgeHours}`. As written the status wiring won't compile against the real codebase.

## Gaps

- NO RE-TAKE / RE-ROLL SEMANTICS DEFINED IN THE BACKEND. §5.1 of the quiz says re-take 'lets the current pet go back into the egg.' But the schema enforces UNIQUE(user_id) on pet, /api/quiz returns 409 already_bonded, and there is NO endpoint to delete/reset a pet. The buried re-take has no API, no migration of bond/exp/streak, no defined data outcome. An engineer hits a wall the moment they build the settings menu.
- NO NAME-INPUT FLOW. `name.given` copy event and `name` column exist, /api/quiz returns a default_name, but there is no endpoint to set/rename the pet. me/home pages don't list a rename action. default_name(archetypeKey) is referenced but never defined (is every mochi pet named '布丁团子'? Then names aren't personal).
- WEATHER IS A FIRST-CLASS COPY/MECHANIC INPUT WITH NO SOURCE. weather (晴/阴/雨/雪/热/冷) drives copy selection, dream_jelly/mochi mechanics, diary openers, and weather.react events — but nothing in the architecture fetches weather. No weather API, no geolocation (mini program needs user authorization for location), no caching table, no fallback when weather is unknown. This is an entire unbuilt subsystem that the 'aliveness' design leans on heavily.
- TIME-OF-DAY / TIMEZONE FOR COPY IS UNDERSPECIFIED. time_band (清晨/白天/傍晚/深夜) and 'user-local 23:00/07:00' sleep windows and '04:00 reset' all need the user's timezone. Schema has tz_offset_minutes DEFAULT 480 but nothing ever SETS it from the client, and WeChat mini programs don't hand you a reliable tz without effort. Every time-gated feature silently assumes UTC+8.
- wisp_moth's ENTIRE PERSONALITY (day-shy/night-active, two sprite forms, polarity-flipped copy) depends on reliable local time-of-day which the gap above leaves unsourced. If tz defaults to UTC+8 for an off-zone user, the headline gimmick creature is broken for them.
- NO EMPTY/ERROR STATES SPECIFIED. What does home render while /api/pet is loading or on network failure? What if a user finishes the quiz but /api/quiz transaction half-fails? What does the client show if assets for the bonded creature's subpackage fail to download? None defined.
- STREAK 'grace' STATE MACHINE IS INCOMPLETE. Core §5 says first miss = freeze (grace), second consecutive miss = floor(streak/2). Schema has streak_grace BOOLEAN but no field tracking consecutive-miss count, and no defined rule for when streak_grace resets to false (after a successful day? after the soft-reset?). The 3-state machine (active/grace/soft-reset) can't be reconstructed from one boolean.
- DAILY GIFT IDEMPOTENCY/ANTI-DOUBLE-CLAIM. checkin is DB-unique-per-day (good) but the gift ROLL is random; if the action_log insert succeeds but the inventory credit fails (or client retries the 200), is the gift re-rolled or replayed? No idempotency key on the gift itself.
- 看医生 COSTS 5 爱心 BUT SICKNESS IS REACHABLE WITH 0 HEARTS. The economy invariant says 'reunion gifts cover medicine' and 'broke player can still 抚摸/玩耍/哄睡 free' — but health only auto-heals at +1/h with satiety>=60 AND cleanliness>=60 AND energy>=50 AND not other constraints. A returning broke player who is SICK with no hearts and an empty larder has a defined-but-very-slow path; the 'reunion care-package gift' that supposedly covers this is mentioned but never specified (what's in it, what triggers it, the table doesn't exist).
- NO DEFINITION OF cap_for_stage(stat) PER-STAT. The stage table gives a single cap per stage (80/85/90/95/100) but bond is 0-1000 and exp is uncapped — the formula `clamp(S - r*Δh, 0, cap_for_stage(stat))` is applied to live stats only; this is implied but never stated, and health's cap interaction with the start-value contradiction (see inconsistencies) is unhandled.

## Scope risks

- COPY BANK IS THE PRODUCT AND THE ESTIMATE IS OPTIMISTIC BY ~2-4x IN CALENDAR TIME. The plan says ~2,000 authored units, ~2-3 days per archetype, one bilingual writer. That is ~150 high-quality, voice-locked, screenshot-worthy lines PER archetype PER 2 days = ~75 publishable lines/day sustained for 20+ days. Realistic rate for lines that must each be funny/in-character/non-repetitive and survive a 30-day repeat audit is closer to 20-30 keepers/day, i.e. 6-8 weeks of dedicated senior copywriting for all 10, not ~3 weeks. And there is ONE such writer assumed. This is the single biggest schedule risk and it is the heart of the product.
- 10 ARCHETYPES FULLY VOICED IS THE WRONG V1 SCOPE given one-pet-per-user. A player only ever sees ONE archetype. Shipping all 10 at gold quality front-loads ~6-8 weeks of copy before a single player can be delighted, while 90% of that copy is invisible to any given user. The design even admits this ('one-pet-per-user means each player only experiences one archetype deeply') but then commits V1 to all 10 anyway.
- SPRITE PIPELINE: 50 'authored' idle sprites understates the real cost. Each authored idle must pass a human curation gate, be re-promptable, quantize to a SHARED 32-color palette with <=4 hue families, AND maintain a consistent silhouette across 5 growth stages of the SAME creature (a recognized-but-matured through-line) — that consistency across stages is exactly what diffusion models are worst at. Expect heavy manual pixel cleanup per sprite. 50 curated, on-model, stage-consistent pixel heroes is weeks of art direction, not a build step.
- THE DERIVED-MOOD CLAIM IS PARTLY FICTION FOR THIS ART. 'sad/sick = desaturate+tint' and 'happy = highlight+sparkle' work for color but NOT for the personality-defining poses the bestiary promises: mochi 'deflates when set down', ember '炸毛→跺脚→离家' multi-pose tantrum, echo_fox 'tail-flick aloof', clay_golem 'turns away'. These are POSE changes, not palette LUTs. The design defers 'eating' as the only authored second pose, but soft-fail states (sulk/hide/reunion) — which are explicitly 'the consequence IS the content' — also need authored poses and are NOT in the V1 art budget. The soft-fail emotional payoff and the art budget contradict.
- WEATHER SUBSYSTEM (unscoped, see gaps) is a non-trivial add: location permission UX, a weather provider, caching, and ~6 weather variants × 10 archetypes of copy. It's woven through the aliveness design but costed nowhere.
- PER-CREATURE SUBPACKAGE LAZY-LOAD on hatch is reasonable but adds WeChat subpackage config + preloadRule keyed off quiz result + a download-progress UX on the hatch screen — none of which is specified, and hatch is the emotional peak moment where a download spinner would kill the magic.

## Abuse vectors

- MULTI-DEVICE / ANONYMOUS-USER FARMING. /api/login falls back to anon-<uuid> when WX creds are absent, and quiz creates a fresh pet + start kit (5 kibble/2 snack/2 soap/20 hearts) per user_id. Nothing binds anon users to a device or rate-limits account creation, so a user can spin unlimited anon accounts for start kits — but since there's no trading/IAP/leaderboard economy in V1, the payoff is low. Still worth a per-IP/device creation cap before any future economy or social feature lands.
- CHECK-IN DAY-BOUNDARY ARBITRAGE VIA tz_offset_minutes. The daily 签到/gift/streak roll over on user-local day, and tz_offset_minutes is client-influenced (and currently never validated). If the client can set or spoof its tz offset, a user can advance their local 'day' to claim multiple daily check-ins/gifts and inflate streaks. The server uses server `now` for decay (good) but local-DATE for daily caps — the local-date computation must use a SERVER-FIXED or validated tz, never a client-sent offset, or the daily economy is farmable.
- 04:00 COUNTER-RESET RACE. feed_count_day etc. reset when daily_reset_on != today. If two actions arrive concurrently (double-tap / retry) around the reset boundary or at the cap, without row-level locking (SELECT FOR UPDATE) on pet_cooldown the counter can be reset/incremented inconsistently, letting a user exceed the 6-meal/4-snack/20-pet daily caps. The spec says 'one DB transaction' but never specifies locking; compute-on-read + action in the same row needs explicit FOR UPDATE.
- COOLDOWN BYPASS VIA CONCURRENT REQUESTS. Cooldowns are checked as `now - last_<verb> < cd` then last_<verb>=now is written. Two simultaneous requests both reading the old last_<verb> both pass the gate before either writes — classic check-then-act race. Without row locking a user can fire N plays/feeds in one burst, bypassing the cooldown and the diminishing-returns/clamp protections partially (clamp still bounds stat totals, but EXP/bond/streak gains and gift rolls can be double-applied).
- CLIENT CLOCK TAMPERING — correctly defended. Δh clamped to [0,720] with server `now`, server timestamps only, client timestamp ignored. Forward/backward phone-clock skew does nothing. This vector is genuinely closed by the design; noted as a PASS, not a hole.
- GROWTH-GATE GRINDING — correctly defended. Stage advance gated on real days_since(created_at) AND bond_gate AND exp, so spamming actions in one sitting can't rush stages. Min-days gate closes the grind. PASS.
- REUNION-GIFT / NEGLECT EXPLOIT. The 'gentle path back' grants reunion EXP bonus + (unspecified) care-package on return after long absence. If a user can repeatedly trigger the >48h LONELY/reunion state to farm reunion bonuses/care-packages, neglect becomes a rewarded strategy. The reunion reward must be rate-limited (e.g. once per N days, or strictly less value than just playing normally) — currently unspecified.

## Aliveness risk

"THE CENTRAL BET IS PLAUSIBLE BUT THE DESIGN AS WRITTEN IS NOT YET ENOUGH, and one structural flaw actively undermines it. The combinatorial math (archetype voice × context slots × anti-repeat rings) genuinely can mask repetition for the high-frequency events IF three things hold that are currently shaky: (1) The MEMORY CLAUSE is the load-bearing illusion — the design correctly identifies it as the highest-ROI trick, but only 5 history flags fire it. After ~2 weeks a daily player will have seen all 5 flag-types repeatedly; the 'it remembers yesterday' effect decays into 'it has 5 canned reactions to my behavior.' To feel alive past week 2 you need history flags that compound (streak-length-aware, bond-tier-aware, 'you always feed me late', 'we've done this 30 days') — i.e. references to ACCUMULATED relationship, not just last-24h flags. The current spec has none of that depth. (2) CONTEXT VARIETY is gated on inputs that are unsourced or unreliable — weather (no provider), time-of-day (tz unset), recent_action (works). Strip out weather and reliable time-of-day and ~half the 'aware of right now' variety the design promises evaporates, collapsing back toward mood_band × streak_tier which a daily player exhausts fast. (3) The DIARY is seeded once/day and is the screenshot line; its freshness rests entirely on the ~1,200-permutation claim, which is real ONLY if every part-pool is actually authored to the stated depth (6 openers × 8 bodies × 5 memory × 5 signoffs PER archetype = the copy-budget risk above). If the writer under-delivers on any one pool, permutations collapse multiplicatively. NET: a well-executed version of this CAN feel alive for the cozy-daily-ritual audience (the bar is QQ宠物/Tamagotchi nostalgia, not conversational AI), but the current design is one or two days of novelty plus a slow fade unless you (a) add relationship-accumulating memory clauses, (b) actually source weather+timezone or cut them and over-invest the freed budget into recent_action/streak/bond context, and (c) treat the diary part-pools as a hard, audited deliverable, not an estimate. The aliveness is buyable but it is NOT yet bought by this spec."

## V1 cut line

"Ship ONE creature, fully, to a 体验版 — not ten. Concretely: (1) ONE archetype end-to-end — pick mochi_pudding or ember_imp (highest-amplitude, most legible 'alive' signal, already gold-drafted in the copy samples). Quiz still runs and still matches across all 10 axes, but in V1 EVERY result maps to the one shipped creature for now, OR (cleaner) ship 3 creatures spanning the corners (mochi clingy, echo_fox aloof, ember explosive — the three already voice-packed in §3) and collapse the quiz to route only among those three. (2) Core loop only: compute-on-read tick + the 6 recovery/care verbs + daily 签到 + daily 心声 + the 5 soft-fail states with SPRITE coverage (idle/happy/sad/sleeping derived; the ONE extra authored pose that matters — sulk/hide — done for just the shipped creatures). (3) DEFER: weather entirely (omit weather predicates; copy still works on time/mood/streak/recent_action), 订阅消息 push (keep the opt-in UI dead-ended or out), codex/图鉴 tab, growth past 'child' (ship egg→baby→child only; teen/adult are weeks-3+ content nobody sees on day 1), seasonal/confession/milestone flavor events, per-creature subpackage lazy-load (3 creatures fit in main package as indexed-palette sheets). (4) Placeholder art is the right call — ship the deterministic-blob placeholders for the unshipped 7 so the quiz/codex don't break, real pixel art for only the 3 shipped. This gives a real, screenshottable, emotionally-complete vertical slice in ~3-4 weeks instead of ~8-10, and lets you VALIDATE the aliveness bet on real users before paying the full 10-creature copy+art bill."

## Top fixes

- FIX THE QUIZ DISTRIBUTION SKEW (highest priority — it breaks the 'fated' promise). Measured over all 4096 answer combinations: puff_seal wins 25%, dream_jelly 20%, while echo_fox/spark_sprite/ember_imp each win ~3% — an 8x imbalance. The answer-space centroid is (attach 1.25, curio 0, express 1.0) and puff_seal sits right on it, acting as a vacuum; the marketing-iconic corner creatures are nearly unreachable. The design's claim of 'distribution not biased' is false. Re-balance: zero out the net attach (+0.21/opt) and net express (+0.17/opt) baselines in the option weights, pull the three 'mid' anchors (puff_seal/sproutling/dream_jelly) AWAY from the origin, and add answer options that reach the negative-attach and high-express corners. Re-run the enumeration until no archetype is below ~6% or above ~16%.
- RECONCILE THE THREE MOOD/STATE/SPRITE VOCABULARIES into one table. Define an explicit map: mood_band (5) + state_flags (7) -> dominant_state -> sprite_file (the art matrix's idle/happy/sad/sleeping + the authored sulk/hide poses) -> animation cue string. Right now /api/action promises animation strings ('refuse_pout','sulk','hide','reunion') that have no authored or derived sprite. Either author those poses (art-budget impact) or redefine the cue vocabulary to only reference sprites that exist.
- ADD RELATIONSHIP-ACCUMULATING MEMORY CLAUSES to the diary, not just 5 last-24h flags. To survive past week 2, the diary needs history clauses keyed to streak length, bond tier, total days known, and behavioral patterns ('你总是很晚才喂我','我们认识30天了'). This is the cheapest insurance for the entire aliveness bet and the current design lacks it.
- RESOLVE OR CUT WEATHER AND TIMEZONE. Either (a) spec a real weather provider + location-permission UX + cache table + tz capture from the client, or (b) cut weather from V1 entirely and set time_band off a single assumed tz, and REINVEST the freed copy budget into recent_action/streak/bond variety. Do not ship a design that leans on weather/time aliveness with no source — wisp_moth and dream_jelly's headline gimmicks silently break otherwise.
- FIX THE health=90 > egg-cap=80 START CONTRADICTION and define cap-per-stat + decay-modifier stacking order explicitly. State that the 4 decay multipliers compose multiplicatively (and whether bond-comfort applies while asleep), and either raise egg health cap to >=90 or lower start health to <=80 so a fresh hatch isn't clamped on first read.
- SPEC THE RE-TAKE AND RENAME ENDPOINTS + DATA SEMANTICS. The UNIQUE(user_id) pet constraint + 409 already_bonded makes the buried re-take impossible to implement as designed. Define: does re-take delete the pet, what happens to bond/exp/streak/inventory, and add the rename endpoint that name.given/the name column already assume.
- REWRITE THE /status WIRING TO THE ACTUAL CODEBASE SHAPES. The cloudpet.ts/cron.ts/http.ts snippets in the tech arch use invented APIs (withDb, Check[] with ok/warn, {group,job,label,maxAgeHours}). The real repo uses CheckFn returning one {id,group,name,status,detail}, query(db,sql), and Job {id,name,schedule,warnAfterMs,failAfterMs}. Match the exemplars in /home/liharr/src/status/web/src/lib/ or the wiring won't compile.
- SPECIFY THE RETURNING-BROKE-SICK RECOVERY PATH CONCRETELY. Define the 'reunion care-package' contents and trigger (it's referenced as covering 看医生's 5-heart cost but the table doesn't exist), so a player who returns SICK with 0 hearts and empty inventory has a guaranteed frictionless recovery — otherwise the 'always reversible, never punishing' core promise has a hole.
